# Important Rules

## Content Rules
1. NEVER create duplicate content pages for different keywords
2. Every page must have unique value and purpose
3. NEVER use AI-generated generic content
4. NEVER keyword stuff
5. Write for humans first, search engines second
6. Every page must satisfy clear search intent
7. Avoid doorway pages that only exist to redirect to hubs

## Design Rules
1. NEVER add a top navigation menu (mobile-first design)
2. NEVER change the dark fintech background (#020617)
3. NEVER switch to light theme
4. NEVER add stock photos or generic images
5. ALWAYS use Inter font
6. ALWAYS use gradient text for H1s
7. ALWAYS use rounded corners (16-32px)
8. ALWAYS include sticky CTA bar on commercial pages
9. ALWAYS use footer-column CSS pattern for footers
10. Preserve existing premium design system

## Legal Rules
1. NEVER claim AI replaces attorneys or professionals
2. NEVER guarantee detection of all hidden fees
3. NEVER guarantee savings or outcomes
4. NEVER create fake testimonials, reviews, or team members
5. ALWAYS disclose AI limitations
6. ALWAYS include "not legal/financial advice" disclaimers
7. ALWAYS be transparent about data handling
8. NEVER claim "same as law firms use" or equivalent
9. NEVER create fictional employees or experts

## SEO Rules
1. NEVER delete existing pages without verification
2. NEVER change canonical URLs without understanding the cluster
3. NEVER let any page become an orphan (no internal links pointing to it)
4. ALWAYS include canonical tags
5. ALWAYS include schema markup
6. ALWAYS optimize title tags and meta descriptions
7. ALWAYS link supporting pages to hub pages
8. Track keyword cannibalization risks between similar pages

## Technical Rules
1. NEVER run build scripts without understanding what they do (many are legacy)
2. ALWAYS use absolute paths for internal links
3. ALWAYS use `noopener noreferrer` for external links
4. ALWAYS include `dns-prefetch` + `preconnect` for hiddenfeeai.com
5. ALWAYS use `loading="lazy"` for images below the fold
6. NEVER duplicate Organization or WebSite schema
7. NEVER add self-review schema
8. ALWAYS verify schema count after changes
