# 🏛️ Authority Page Strategy

## How Pages Connect Into Topic Authority

This document defines the six page types used on DetectHiddenFees.com, how they interlink, and how they collectively build AI-recognized topic authority.

---

## Page Type 1: Pillar Pages

**Purpose:** The definitive page for one core entity. The single most authoritative page on the site for that topic.

**Quantity:** 1 per entity (9 total)
**Target length:** 2,500-4,000 words
**Target content score:** 80+

**Characteristics:**
- Answers the primary entity question comprehensively
- Links to ALL supporting pages in the entity cluster
- Contains original examples, data, and frameworks
- Serves as the canonical URL for the entity
- Updated regularly (every 90 days minimum)
- Includes FAQPage schema with entity-specific questions

**Example:**
- `ai-contract-review.html` — "How AI Contract Review Works: Find Hidden Fees Before You Sign"
- `ai-bill-analyzer.html` — "AI Bill Analysis: How Artificial Intelligence Finds Billing Errors"

**Internal linking role:**
- Links OUT to: all supporting pages, knowledge pages, tool pages in the entity cluster
- Linked FROM: homepage, hub pages, all supporting pages in the cluster

---

## Page Type 2: Supporting Pages

**Purpose:** Deeper coverage of a specific aspect, subtopic, or angle within an entity cluster.

**Quantity:** 3-15 per entity
**Target length:** 1,200-2,500 words
**Target content score:** 70+

**Characteristics:**
- Covers a distinct subtopic of the pillar entity
- Links back to the pillar page
- Has a specific, narrow question it answers
- Contains unique examples relevant to the subtopic
- Often industry-specific or use-case-specific

**Subtypes:**
- **Industry-specific guides** (e.g., "Hidden HVAC Contractor Fees")
- **Comparison pages** (e.g., "AI Contract Review vs. Lawyer Review")
- **Methodology pages** (e.g., "How AI Analyzes Contract Language")
- **Use case pages** (e.g., "AI Review for Employment Contracts")

**Example:**
- `hidden-hvac-contractor-fees.html` — supporting "Hidden Fees" pillar
- `ai-contract-review-vs-lawyer-review.html` — supporting "AI Contract Review" pillar
- `ai-employment-contract-review.html` — supporting "AI Agreement Analyzer" pillar

**Internal linking role:**
- Links OUT to: pillar page, related supporting pages, tool pages
- Linked FROM: pillar page, hub pages, related supporting pages

---

## Page Type 3: Knowledge Pages

**Purpose:** Educational deep-dives that explain concepts, methodology, or industry context.

**Quantity:** 3-8 per entity
**Target length:** 1,500-3,000 words
**Target content score:** 75+

**Characteristics:**
- Educational, not transactional
- Explains how something works or why something matters
- Often methodology, accuracy, or transparency-focused
- Builds E-E-A-T (Experience, Expertise, Authoritativeness, Trustworthiness)
- Heavy on citations and references

**Subtypes:**
- **Methodology pages** (e.g., "AI Analysis Methodology")
- **Accuracy pages** (e.g., "AI Accuracy and Limitations")
- **Educational guides** (e.g., "How Automatic Renewal Clauses Work")
- **Concept explainers** (e.g., "What Is Contract Risk Analysis?")

**Example:**
- `ai-analysis-methodology.html`
- `ai-accuracy-and-limitations.html`
- `arbitration-clauses-explained.html`
- `contract-terms-glossary.html`

**Internal linking role:**
- Links OUT to: pillar page, related supporting pages, glossary pages
- Linked FROM: pillar page, supporting pages that reference the methodology/concept

## Page Type 4: Glossary Pages

**Purpose:** Reference pages defining key terms, clauses, and concepts.

**Quantity:** 2-5 total
**Target length:** 1,000-2,000 words (term listings with definitions)
**Target content score:** 65+

**Characteristics:**
- Defines industry terminology in plain English
- Links to deeper pages for each term
- Structured for quick reference (definition format)
- Useful for both humans and AI term extraction
- Earns "definition" featured snippets

**Example:**
- `contract-terms-glossary.html` — "Contract Terms Glossary: 50+ Legal Terms Explained in Plain English"
- Future: `hidden-fee-glossary.html` — "Hidden Fee Dictionary: Every Fee Type Explained"

**Internal linking role:**
- Links OUT to: supporting pages for each defined term
- Linked FROM: supporting pages, knowledge pages (as "Learn more about this term")

---

## Page Type 5: Tool Pages

**Purpose:** Interactive or action-oriented pages where users DO something.

**Quantity:** 5-15 total
**Target length:** 500-1,200 words (tool-focused, less text-heavy)
**Target content score:** 65+

**Characteristics:**
- Primary purpose is action (upload, scan, check, calculate)
- Clear CTA above the fold
- Explains what the tool does and how it works
- Links to educational pages for "how this works" context
- May include interactive elements (calculators, upload forms)

**Subtypes:**
- **Upload pages** (e.g., "analyze-my-contract.html")
- **Checker/scanner pages** (e.g., "ai-bill-checker.html")
- **Calculator pages** (e.g., "bill-savings-calculator.html")

**Example:**
- `analyze-my-contract.html`
- `ai-bill-checker.html`
- `bill-savings-calculator.html`
- `check-my-fees.html`

**Internal linking role:**
- Links OUT to: methodology page (how the tool works), supporting guides
- Linked FROM: pillar page, supporting pages (as "Try our [tool]" CTA), homepage

---

## Page Type 6: Research Pages

**Purpose:** Original research, data reports, and proprietary analysis that establishes primary-source authority.

**Quantity:** 2-5 initially, growing over time
**Target length:** 2,000-5,000 words
**Target content score:** 85+

**Characteristics:**
- Contains original data not found elsewhere
- Methodology is transparent and documented
- Cited by other sites and AI engines
- Updates annually or as data changes
- Primary link-building asset

**Example:**
- `consumer-fee-trends-report.html` — "Annual Consumer Hidden Fee Trends Report"
- Future: "AI Fee Detection Accuracy Benchmark Study"
- Future: "State-by-State Hidden Fee Regulation Report"

**Internal linking role:**
- Links OUT to: methodology page, entity pillar pages
- Linked FROM: homepage, pillar pages (as "Our research shows..."), press/media pages

## Page Creation Decision Tree

When considering a new page, ask:

1. **Is this a PILLAR?** → Is this the #1 most important page for an entity? If yes, only create if the entity lacks a pillar entirely.

2. **Is this a SUPPORTING page?** → Does it cover a distinct subtopic or use case? Is there at least 60% unique content vs. existing pages? If yes, create using the authority-page-template.

3. **Is this a KNOWLEDGE page?** → Does it explain methodology, build E-E-A-T, or educate on a concept? Is it referenced by multiple other pages? If yes, create.

4. **Is this a GLOSSARY page?** → Will it define 20+ terms with links to deeper content? If yes, create.

5. **Is this a TOOL page?** → Does it enable a specific action (upload, check, calculate)? Is there a working tool behind it? If no working tool, don't create an empty tool page.

6. **Is this a RESEARCH page?** → Does it contain original data or analysis? Can we cite specific sources and methodology? If yes, these are the highest-authority pages we can create.

---

## Authority Building Sequence

When building out a new entity, create pages in this order:

1. **PILLAR first** — The entity needs a home. Create the pillar page before any supporting content.
2. **2-3 SUPPORTING pages** — Establish depth with key subtopics and comparisons.
3. **1-2 KNOWLEDGE pages** — Build E-E-A-T with methodology and educational content.
4. **TOOL pages** — Add action-oriented pages once the educational foundation exists.
5. **GLOSSARY pages** — Add as the cluster grows and terminology needs defining.
6. **RESEARCH pages** — Add original research once the entity has sufficient depth to support it.

**Never:** Create tool pages without educational content backing them. Never create supporting pages without a pillar. Never create research without methodology transparency.

---

> **Pages don't build authority in isolation. They build authority through connection — each page reinforcing the pillar, the pillar connecting the cluster, the cluster defining the entity.**
